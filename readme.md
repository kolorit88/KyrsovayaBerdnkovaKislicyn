## Таблица соответствия требованиям

| Требование                                               | Где в теме реализуется                                                                                                                                                                                                                       | Какой endpoint/сценарий                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| :------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **1. Роли и доступы (минимум 2 роли)**                   | Реализовано три роли: **Администратор**, **Оператор**, **Пользователь**. Доступ к эндпоинтам разграничен на основе ролей с помощью Spring Security.                                                                                          | • `POST /api/admin/merch` (только ADMIN)<br>• `PUT /api/admin/merch/{id}` (только ADMIN)<br>• `GET /api/operator/orders` (ADMIN, OPERATOR)<br>• `PUT /api/operator/orders/{id}/status` (ADMIN, OPERATOR)<br>• `POST /api/user/orders` (ADMIN, OPERATOR, USER)<br>• `GET /api/user/cart` (ADMIN, OPERATOR, USER)                                                                                                                                                                                                                                                                                                                                                       |
| **2. Сущности и модель (минимум 5 сущностей и 1 связь)** | Выделено 6 ключевых сущностей. Связи: `Order` (Заказ) связан с `User` (Пользователь) и `OrderItem` (Пункт заказа). `Merchandise` (Товар) связан с `Category` (Категория).                                                                    | **Сущности:** <br>1. `User` (пользователь)<br>2. `Role` (роль)<br>3. `Merchandise` (товар)<br>4. `Category` (категория товаров)<br>5. `Order` (заказ)<br>6. `OrderItem` (позиция в заказе)<br><br>**Связи:** `Order` (1) <--> (M) `OrderItem`.                                                                                                                                                                                                                                                                                                                                                                                                                        |
| **3. Транзакционный сценарий**                           | **Оплата заказа.** Важно атомарно изменить статус заказа и списать средства/обновить остаток товара (если есть), чтобы избежать двойного списания или оплаты несуществующего товара.                                                         | `POST /api/user/orders/{id}/pay` <br> Внутри сервиса используется `@Transactional`. В случае ошибки на любом этапе — полный откат операции. Защита от гонок реализуется через оптимистичные или пессимистичные блокировки (например, `@Version` в сущности `Merchandise`).                                                                                                                                                                                                                                                                                                                                                                                            |
| **4. Кэшируемый сценарий (Redis)**                       | **Отправка меню (каталога товаров) пользователю.** Меню — самые часто запрашиваемые, но редко изменяемые данные (администратором). Кэширование в Redis снижает нагрузку на БД и ускоряет загрузку страницы для пользователей.                | `GET /api/menu` <br> При первом запросе данные читаются из БД и сохраняются в Redis (аннотация `@Cacheable`). При изменении товара администратором (`PUT /api/admin/merch/{id}`) кэш инвалидируется (`@CacheEvict`).                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| **5. Фоновый сценарий**                                  | **@Scheduled: Автопереход статусов.** Заказы со статусом "Ожидает оплаты" дольше 30 минут автоматически переводятся в статус "Отменен" (тайм-аут).<br>                                                                                       | Метод в сервисе заказов с аннотацией `@Scheduled(cron = "0 */5 * * * *")` запускается каждые 5 минут и проверяет просроченные заказы.<br>                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **6. Безопасность**                                      | Естественное требование: все действия привязаны к роли. Аутентификация по логину/паролю (Spring Security). Пароли пользователей хешируются (BCrypt). Администратор и оператор создаются через специальные скрипты или панель администратора. | • Регистрация: `POST /api/auth/register`<br>• Вход: `POST /api/auth/login` (возвращает JWT токен).<br>Все остальные эндпоинты (кроме `/api/menu`) требуют наличия в запросе валидного JWT токена с соответствующими ролями.                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **7. Тестируемость**                                     | Ключевые сценарии покрыты интеграционными тестами с использованием `Testcontainers` (поднятие PostgreSQL и Redis в контейнерах).                                                                                                             | • **Тест разграничения прав:** Попытка оператора создать товар (`POST /api/admin/merch`) должна вернуть `403 Forbidden`.<br>• **Тест атомарности транзакции:** Симуляция двух одновременных запросов на оплату одного и того же заказа. В результате статус заказа должен измениться только один раз.<br>• **Тест отправки заказа:** Проверка, что после `POST /api/user/orders` заказ корректно сохраняется в БД, а товары из корзины пользователя удаляются.                                                                                                                                                                                                        |
| **8. Наблюдаемость**                                     | Сбор метрик и логов для мониторинга работоспособности и бизнес-показателей (с использованием Micrometer, Prometheus, ELK).                                                                                                                   | **Метрики:**<br>• **Бизнес-метрики (Prometheus):** количество заказов за месяц, средний чек, популярные позиции, причины отмены заказа (считаются на основе данных из БД и экспортируются через отдельный эндпоинт).<br>• **Технические метрики (Micrometer):** время ответа сервера (`http_server_requests_seconds`), время загрузки страницы (можно измерить на фронтенде), количество активных сессий.<br>• **Трейсинг (Spring Cloud Sleuth + Zipkin):** отслеживание цепочки вызовов микросервисов (если они будут) или запросов к БД для выявления узких мест.<br>• **Логи (SLF4J):** структурированные логи (JSON) для централизованного сбора и поиска ошибок. |
| **9. Frontend-покрытие (Vue 3 + TypeScript)**            | Реализованы все ключевые UI-сценарии для каждой роли.                                                                                                                                                                                        | • **Пользователь:** Страница "Меню" (список), "Карточка товара" (детали), "Корзина" (создание заказа), "Личный кабинет" (история заказов).<br>• **Оператор:** "Панель оператора" (список заказов с фильтрацией), страница "Детали заказа" (изменение статуса).<br>• **Администратор:** "Панель администратора" (управление товарами: создание, редактирование, удаление; управление категориями).                                                                                                                                                                                                                                                                     |

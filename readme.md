## Предлагаемые темы работ

### 1. Система записи в поликлинику
**Бизнес-контекст:** сервис для управления расписанием врачей и онлайн-записью пациентов.

**Роли и права:**
1. `ADMIN` управляет справочниками, врачами и слотами.
2. `DOCTOR` видит свои записи, подтверждает и завершает приемы.
3. `PATIENT` ищет слоты, записывается и отменяет запись.

**Обязательные сущности:**
1. `UserAccount` (логин, пароль, роль, статус).
2. `DoctorProfile` (ФИО, специализация, кабинет).
3. `PatientProfile` (ФИО, дата рождения, контакты).
4. `Specialization` (кардиолог, терапевт и т.д.).
5. `ScheduleSlot` (врач, время начала/конца, статус).
6. `Appointment` (слот, пациент, статус, комментарий).

**Ключевые бизнес-правила:**
1. На один `ScheduleSlot` может быть не более одной активной записи.
2. Запись доступна только на будущие слоты со статусом `OPEN`.
3. Врач не может завершить прием, который не относится к нему.
4. После отмены запись не может быть завершена.
5. История изменения статусов приема должна сохраняться.

**Обязательные пользовательские сценарии (MVP):**
1. Админ создает врача, специализацию и сетку слотов на день.
2. Пациент находит слот по дате и специализации, затем записывается.
3. Пациент отменяет запись.
4. Врач просматривает расписание и завершает прием.

**Минимальный API-контур:**
1. CRUD для `DoctorProfile`, `PatientProfile`, `Specialization`.
2. CRUD для `ScheduleSlot` (с фильтрами по врачу и дате).
3. `POST /appointments` создание записи.
4. `PATCH /appointments/{id}/cancel` отмена записи.
5. `PATCH /appointments/{id}/complete` завершение приема.
6. `GET /appointments` с фильтрами по пациенту, врачу и статусу.

**Критерии приемки:**
1. Невозможна двойная запись в один и тот же слот.
2. Ограничения ролей подтверждаются тестами.
3. История статусов приема доступна через API.

### 2. Платформа для онлайн-аукционов
**Бизнес-контекст:** площадка, где продавцы публикуют лоты, а покупатели делают ставки.

**Роли и права:**
1. `SELLER` создает и публикует лоты.
2. `BUYER` делает ставки и просматривает историю торгов.
3. `MODERATOR` (опционально) блокирует некорректные лоты.

**Обязательные сущности:**
1. `UserAccount`.
2. `Lot` (название, описание, стартовая цена, шаг ставки, время окончания, статус).
3. `Bid` (лот, пользователь, сумма, время).
4. `LotStatusHistory`.
5. `AuctionResult` (победитель, финальная цена, время закрытия).

**Ключевые бизнес-правила:**
1. Ставка принимается только если лот в статусе `ACTIVE`.
2. Новая ставка должна быть больше текущей максимальной минимум на шаг.
3. После закрытия лота новые ставки запрещены.
4. Продавец не может делать ставки на собственный лот.
5. Победителем становится автор максимальной ставки.

**Обязательные пользовательские сценарии (MVP):**
1. Продавец создает черновик лота и публикует его.
2. Покупатели делают ставки в течение активной фазы.
3. Система закрывает лот и фиксирует победителя.
4. Пользователь просматривает полную историю ставок по лоту.

**Минимальный API-контур:**
1. CRUD для лотов.
2. `POST /lots/{lotId}/publish`.
3. `POST /lots/{lotId}/bids`.
4. `GET /lots/{lotId}/bids`.
5. `POST /lots/{lotId}/close`.
6. `GET /lots/{lotId}/result`.

**Критерии приемки:**
1. Ставка ниже допустимого порога отвергается.
2. После закрытия лота ставки не принимаются.
3. Победитель и финальная цена детерминированы и доступны через API.

### 3. Сервис продажи билетов в кино/театр
**Бизнес-контекст:** покупка мест на сеанс с контролем доступности каждого места.

**Роли и права:**
1. `MANAGER` управляет мероприятиями, залами и сеансами.
2. `CUSTOMER` бронирует и покупает билеты.

**Обязательные сущности:**
1. `Event` (фильм/спектакль).
2. `Hall`.
3. `Seat` (ряд, номер, категория цены).
4. `Session` (мероприятие + зал + время).
5. `Booking` (места, пользователь, статус, срок брони).
6. `Ticket` (номер, цена, статус).

**Ключевые бизнес-правила:**
1. Одно место на конкретный сеанс не может быть продано дважды.
2. Бронирование имеет срок жизни и после истечения освобождает места.
3. Покупка возможна только для действующей брони.
4. Отмена покупки меняет статус билета, но не удаляет историю.
5. Список мест отражает фактические состояния `FREE/BOOKED/SOLD`.

**Обязательные пользовательские сценарии (MVP):**
1. Менеджер создает мероприятие, зал и сеанс.
2. Пользователь запрашивает карту мест сеанса.
3. Пользователь бронирует несколько мест одним запросом.
4. Пользователь подтверждает покупку и получает билеты.

**Минимальный API-контур:**
1. CRUD для `Event`, `Hall`, `Session`.
2. `GET /sessions/{id}/seats`.
3. `POST /sessions/{id}/bookings`.
4. `POST /bookings/{id}/confirm`.
5. `POST /bookings/{id}/cancel`.
6. `GET /users/{id}/tickets`.

**Критерии приемки:**
1. Конфликтные покупки одного места корректно обрабатываются.
2. Карта мест после покупки отражает обновленное состояние.
3. История бронирований и билетов доступна пользователю.

### 4. Система управления библиотекой
**Бизнес-контекст:** учет книжного фонда, выдач и возвратов экземпляров.

**Роли и права:**
1. `LIBRARIAN` управляет каталогом и выдачами.
2. `READER` просматривает каталог, берет и возвращает книги.

**Обязательные сущности:**
1. `BookTitle` (название, автор, ISBN).
2. `BookCopy` (инвентарный номер, статус, ссылка на `BookTitle`).
3. `Reader`.
4. `Loan` (выдача, срок возврата, статус).
5. `Fine` (просрочка/повреждение, сумма, статус оплаты).
6. `LoanHistory`.

**Ключевые бизнес-правила:**
1. Экземпляр в статусе `LOANED` нельзя выдать повторно.
2. Возврат закрывает активную выдачу и обновляет статус экземпляра.
3. Просрочка формирует штраф.
4. У одного читателя ограничено число активных выдач.
5. Все операции выдачи и возврата должны быть трассируемыми.

**Обязательные пользовательские сценарии (MVP):**
1. Библиотекарь добавляет название и несколько экземпляров книги.
2. Читатель получает книгу на руки.
3. Читатель возвращает книгу в срок и с просрочкой.
4. Библиотекарь получает список активных просрочек.

**Минимальный API-контур:**
1. CRUD для `BookTitle`, `BookCopy`, `Reader`.
2. `POST /loans/issue`.
3. `POST /loans/{id}/return`.
4. `GET /loans/overdue`.
5. `GET /readers/{id}/loans`.
6. `GET /fines`.

**Критерии приемки:**
1. Нельзя выдать занятый экземпляр.
2. Возврат корректно меняет статусы выдачи и экземпляра.
3. Просрочки и штрафы рассчитываются и отдаются через API.

### 5. Сервис аренды автомобилей
**Бизнес-контекст:** управление автопарком и оформлением аренды по периодам.

**Роли и права:**
1. `FLEET_MANAGER` управляет автомобилями, тарифами и окнами обслуживания.
2. `CLIENT` ищет доступные авто и создает аренду.

**Обязательные сущности:**
1. `Car` (VIN, номер, класс, статус).
2. `Tariff` (базовая цена, цена за сутки, ограничения).
3. `ClientProfile`.
4. `RentalOrder` (период, авто, клиент, цена, статус).
5. `MaintenanceWindow`.
6. `RentalStatusHistory`.

**Ключевые бизнес-правила:**
1. Для одного авто запрещены пересекающиеся периоды аренды.
2. Авто в окне обслуживания недоступно для бронирования.
3. Стоимость аренды рассчитывается по тарифу и длительности.
4. Завершенная аренда фиксирует фактическое время возврата.
5. Отмена аренды запрещена после фактической выдачи авто.

**Обязательные пользовательские сценарии (MVP):**
1. Менеджер добавляет авто, тариф и окно обслуживания.
2. Клиент ищет доступные авто на заданные даты.
3. Клиент оформляет аренду.
4. Менеджер переводит аренду в статусы `ISSUED` и `COMPLETED`.

**Минимальный API-контур:**
1. CRUD для `Car`, `Tariff`, `MaintenanceWindow`.
2. `GET /cars/available?from=&to=`.
3. `POST /rentals`.
4. `PATCH /rentals/{id}/issue`.
5. `PATCH /rentals/{id}/complete`.
6. `GET /clients/{id}/rentals`.

**Критерии приемки:**
1. Невозможна аренда недоступного автомобиля.
2. Цена заказа воспроизводима и объяснима по данным тарифа.
3. История статусов аренды доступна через API.

### 6. Биржа фриланса
**Бизнес-контекст:** платформа для публикации задач, откликов и заключения контрактов.

**Роли и права:**
1. `CUSTOMER` публикует задачи и выбирает исполнителя.
2. `FREELANCER` оставляет отклики и сдает результаты.
3. `MODERATOR` блокирует нарушения и спорные проекты.

**Обязательные сущности:**
1. `Project`.
2. `Proposal`.
3. `Contract`.
4. `Milestone`.
5. `Submission`.
6. `Review`.
7. `Dispute`.

**Ключевые бизнес-правила:**
1. По одному проекту может быть только один активный контракт.
2. Нельзя выбрать исполнителя без существующего отклика.
3. Каждый этап (`Milestone`) имеет дедлайн и собственный статус.
4. Закрытие проекта возможно только после закрытия всех этапов.
5. Отзыв можно оставить только после завершенного контракта.

**Обязательные пользовательские сценарии (MVP):**
1. Заказчик публикует проект.
2. Несколько исполнителей подают отклики.
3. Заказчик выбирает отклик и формирует контракт.
4. Исполнитель сдает этап, заказчик принимает или отправляет на доработку.
5. После завершения проекта стороны оставляют отзывы.

**Минимальный API-контур:**
1. CRUD для `Project`.
2. `POST /projects/{id}/proposals`.
3. `POST /projects/{id}/select-proposal`.
4. CRUD для `Milestone`.
5. `POST /milestones/{id}/submissions`.
6. `POST /contracts/{id}/close`.
7. `POST /contracts/{id}/reviews`.

**Критерии приемки:**
1. Ролевые ограничения на все ключевые действия соблюдаются.
2. Проект не может перейти в `COMPLETED` при незакрытых этапах.
3. Связка проект -> контракт -> этапы -> отзывы корректно читается через API.

### 7. Система управления курсами
**Бизнес-контекст:** платформа для обучения с курсами, заданиями и оценками.

**Роли и права:**
1. `TEACHER` управляет структурой курса и проверкой работ.
2. `STUDENT` записывается на курс, отправляет решения.
3. `ADMIN` (опционально) модерирует пользователей и публикации.

**Обязательные сущности:**
1. `Course`.
2. `Module`.
3. `Lesson`.
4. `Enrollment`.
5. `Assignment`.
6. `Submission`.
7. `Grade`.

**Ключевые бизнес-правила:**
1. Доступ к материалам курса имеют только записанные студенты.
2. Модуль и урок имеют порядковый номер внутри курса/модуля.
3. Работу можно сдать только по активному заданию.
4. Оценка выставляется только преподавателем курса.
5. Прогресс студента считается по завершенным заданиям.

**Обязательные пользовательские сценарии (MVP):**
1. Преподаватель создает курс с модулями и уроками.
2. Студент записывается на курс.
3. Студент отправляет домашнюю работу.
4. Преподаватель проверяет работу и выставляет оценку.
5. Студент просматривает текущий прогресс по курсу.

**Минимальный API-контур:**
1. CRUD для `Course`, `Module`, `Lesson`, `Assignment`.
2. `POST /courses/{id}/enroll`.
3. `POST /assignments/{id}/submissions`.
4. `POST /submissions/{id}/grade`.
5. `GET /courses/{id}/progress`.
6. `GET /students/{id}/grades`.

**Критерии приемки:**
1. Студент без `Enrollment` не видит контент курса.
2. Логика оценивания и подсчета прогресса соответствует данным БД.
3. Полная история отправок и оценивания доступна через API.

### 8. Складской учет
**Бизнес-контекст:** учет остатков, приходов, списаний и перемещений между складами.

**Роли и права:**
1. `WAREHOUSE_OPERATOR` оформляет движения товара.
2. `PROCUREMENT_MANAGER` управляет поставщиками и заявками на закупку.

**Обязательные сущности:**
1. `Warehouse`.
2. `Product`.
3. `StockBalance`.
4. `StockMovement`.
5. `Supplier`.
6. `PurchaseOrder`.
7. `PurchaseOrderItem`.

**Ключевые бизнес-правила:**
1. Остаток по товару на складе не может стать отрицательным.
2. Перемещение между складами должно менять остатки в обоих местах.
3. Приход возможен только на существующий склад и товар.
4. Списание возможно только при достаточном остатке.
5. Каждое движение фиксируется в истории с причиной операции.

**Обязательные пользовательские сценарии (MVP):**
1. Оператор оформляет приход от поставщика.
2. Оператор выполняет расход товара.
3. Оператор делает перемещение между складами.
4. Менеджер создает заявку на закупку при низком остатке.

**Минимальный API-контур:**
1. CRUD для `Warehouse`, `Product`, `Supplier`.
2. `POST /stock/inbound`.
3. `POST /stock/outbound`.
4. `POST /stock/transfer`.
5. `GET /stock/balances`.
6. CRUD для `PurchaseOrder`.

**Критерии приемки:**
1. Инвариант неотрицательных остатков не нарушается.
2. История движений согласована с фактическими остатками.
3. Операции перемещения атомарно отражаются на двух складах.

### 9. Сервис вопросов и ответов
**Бизнес-контекст:** публикация вопросов, ответов и оценка полезности контента.

**Роли и права:**
1. `USER` создает вопросы/ответы и голосует.
2. `MODERATOR` скрывает и возвращает контент по правилам площадки.

**Обязательные сущности:**
1. `Question`.
2. `Answer`.
3. `Tag`.
4. `Vote`.
5. `UserProfile`.
6. `ModerationAction`.

**Ключевые бизнес-правила:**
1. Один пользователь может иметь один активный голос по конкретному объекту.
2. Вопрос и ответ имеют статус публикации (`VISIBLE/HIDDEN/DELETED`).
3. Принять ответ может только автор вопроса.
4. Рейтинг вопроса и ответа рассчитывается по голосам.
5. Скрытый модератором контент недоступен обычным пользователям.

**Обязательные пользовательские сценарии (MVP):**
1. Пользователь публикует вопрос с тегами.
2. Другой пользователь публикует ответ.
3. Пользователи голосуют за вопрос и ответы.
4. Автор вопроса отмечает принятый ответ.
5. Модератор скрывает некорректный контент.

**Минимальный API-контур:**
1. CRUD для `Question`, `Answer`, `Tag`.
2. `POST /questions/{id}/votes`.
3. `POST /answers/{id}/votes`.
4. `POST /questions/{id}/accept-answer/{answerId}`.
5. `POST /moderation/hide`.
6. `GET /questions?tag=&sort=`.

**Критерии приемки:**
1. Повторное голосование в обход правила уникальности невозможно.
2. Рейтинг объектов корректно пересчитывается.
3. Модераторские действия видны в журнале модерации.

### 10. Трекер расходов с совместным доступом
**Бизнес-контекст:** учет личных и общих расходов для семьи или команды.

**Роли и права:**
1. `WALLET_OWNER` создает кошельки и управляет участниками.
2. `WALLET_MEMBER` добавляет операции в разрешенные кошельки.

**Обязательные сущности:**
1. `Wallet`.
2. `WalletMember`.
3. `Category`.
4. `Transaction`.
5. `Budget`.
6. `SplitExpense`.
7. `RecurringRule`.

**Ключевые бизнес-правила:**
1. Баланс кошелька вычисляется на основе операций, а не хранится вручную.
2. Для операции обязательно указывается тип (`INCOME`/`EXPENSE`) и категория.
3. Участник может изменять только собственные операции, если не владелец.
4. Бюджет задается на период и сравнивается с фактическими тратами.
5. Разделенный расход должен корректно распределяться между участниками.

**Обязательные пользовательские сценарии (MVP):**
1. Владелец создает кошелек и приглашает участников.
2. Участники добавляют доходы и расходы по категориям.
3. Пользователь формирует отчет за период.
4. Пользователь создает правило регулярного платежа.
5. Владелец просматривает отклонения от бюджета.

**Минимальный API-контур:**
1. CRUD для `Wallet`, `Category`, `Transaction`, `Budget`.
2. `POST /wallets/{id}/members`.
3. `POST /transactions/split`.
4. `GET /reports/period`.
5. CRUD для `RecurringRule`.
6. `GET /wallets/{id}/balance`.

**Критерии приемки:**
1. Баланс и отчеты совпадают с фактическими транзакциями.
2. Права доступа к общему кошельку соблюдаются.
3. Бюджетные показатели и разбивка по категориям доступны через API.

## Требования соответствия для тем, придуманных студентами
Если студент предлагает свою тему (не из списка выше), тема допускается только при соблюдении требований ниже.

## Обязательный технологический стек курсовой

1. **Backend язык и фреймворк:** `Kotlin` + `Spring Boot`.
2. **Frontend язык и фреймворк:** `Vue 3` + `TypeScript`.
3. **Backend архитектура:** слои `Controller -> Service -> Repository`, DTO, валидация входа (`@Valid`), единый формат ошибок (`@ControllerAdvice` / `@ExceptionHandler`).
4. **API:** REST (`GET/POST/PUT/DELETE`) + документация `OpenAPI/Swagger` (SpringDoc).
5. **БД:** `PostgreSQL` + `Spring Data JPA/Hibernate`.
6. **Миграции:** `Flyway` **или** `Liquibase` (один инструмент на проект, обязательно).
7. **Связи данных:** минимум одна нетривиальная связь (`OneToMany` или `ManyToMany`) и корректная загрузка данных (без грубого N+1 в ключевых сценариях).
8. **Безопасность:** `Spring Security` + `JWT` + роли (`USER/ADMIN` или эквивалент по теме).
9. **Тестирование backend:** unit + интеграционные тесты через `Testcontainers` (PostgreSQL).
10. **Контейнеризация:** отдельные `Dockerfile` для backend и frontend + общий `docker-compose`.
11. **Кэширование:** `Redis` + применение `@Cacheable` в обоснованном read-сценарии.
12. **Асинхронность:** минимум один практический сценарий через `@Async` и/или `@Scheduled`.
13. **CI/CD:** `GitHub Actions` для backend и frontend (`build + test` и lint/static checks).
14. **Наблюдаемость:** `Spring Actuator` (`/actuator/health`, `/actuator/metrics`).

## Общий чеклист приемки курсовой
1. **Запуск одной командой:** `docker-compose up` поднимает `backend + frontend + остальное`.
2. **БД и миграции:** схема поднимается автоматически через Flyway.
3. **API-контракт:** OpenAPI/Swagger актуален и соответствует реализации.
4. **Frontend-часть:** реализована на `Vue 3 + TypeScript`, покрывает ключевые пользовательские сценарии темы.
5. **Безопасность:** JWT + роли реально используются и ограничивают действия и на backend, и в UI.
6. **Тесты backend:** есть unit-тесты и интеграционные тесты с Testcontainers.
7. **CI:** GitHub Actions pipeline проходит для backend и frontend (`build + test` минимум).
8. **Кэш и async:** реализованы и продемонстрированы в предметном сценарии.
9. **Наблюдаемость:** подключен Actuator, доступны health/metrics.
10. **Архитектурная чистота backend:** нет бизнес-логики в контроллерах; слойность и DTO соблюдены.
